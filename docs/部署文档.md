# 智能客服系统部署文档

## 一、系统要求

### 硬件要求

| 环境 | CPU | 内存 | 存储 |
|------|-----|------|------|
| 开发环境 | 2核+ | 4GB+ | 10GB+ |
| 生产环境 | 4核+ | 8GB+ | 50GB+ |

### 软件要求

- Python 3.11+
- Docker 20.10+（容器化部署）
- Docker Compose 2.0+（容器化部署）
- LLM API 密钥（三选一，推荐 DeepSeek）：
  - **DeepSeek**：[platform.deepseek.com](https://platform.deepseek.com) 免费注册，国内直连，¥2/百万 token
  - **智谱 GLM**：[open.bigmodel.cn](https://open.bigmodel.cn) 免费注册，国内直连
  - **OpenAI**：[platform.openai.com](https://platform.openai.com) 需要 VPN

## 二、本地部署流程

### 步骤1：进入项目目录

```bash
cd 8.LangChain智能客服系统
```

### 步骤2：创建虚拟环境

```bash
# 创建虚拟环境
python -m venv venv

# 激活虚拟环境
# Windows:
venv\Scripts\activate
# Linux/Mac:
source venv/bin/activate
```

### 步骤3：安装依赖

```bash
pip install -r requirements.txt
```

> 首次运行时会自动下载 Embedding 模型 `BAAI/bge-small-zh-v1.5`（约 95MB）。

### 步骤4：配置环境变量

```bash
# ========== 方案一：DeepSeek（推荐，国内直连）==========
export OPENAI_API_KEY="sk-你的DeepSeek密钥"
# OPENAI_BASE_URL 默认为 https://api.deepseek.com/v1，无需额外设置

# ========== 方案二：智谱 GLM（国内直连）==========
# export OPENAI_API_KEY="你的智谱密钥"
# export OPENAI_BASE_URL="https://open.bigmodel.cn/api/paas/v4"
# export PRIMARY_MODEL="glm-4-flash"

# ========== 方案三：OpenAI（需 VPN）==========
# export OPENAI_API_KEY="sk-你的OpenAI密钥"
# export OPENAI_BASE_URL="https://api.openai.com/v1"
# export PRIMARY_MODEL="gpt-4-turbo"

# ========== 可选：HuggingFace 镜像（国内加速模型下载）==========
export HF_ENDPOINT="https://hf-mirror.com"
```

### 步骤5：运行系统

```bash
# 流式 CLI 交互对话
python main.py

# 演示模式（自动运行 6 个测试用例）
python main.py --demo

# 启动 HTTP API 服务
python main.py --api --host 0.0.0.0 --port 8000
# 访问 http://localhost:8000/docs 查看 Swagger 接口文档
```

## 四、生产环境部署建议

### 1. API 密钥管理

- 使用环境变量或密钥管理服务（如 AWS Secrets Manager、Azure Key Vault）
- 定期轮换 API 密钥
- 限制 API 密钥的权限范围和 IP 白名单

### 2. 高可用部署

```yaml
# docker-compose.yml 高可用配置
services:
  customer-service-bot:
    deploy:
      replicas: 3
      resources:
        limits:
          cpus: '2'
          memory: 4G
    restart: always
```

### 3. Embedding 模型预热

Docker 镜像构建时预下载 Embedding 模型，避免容器启动时下载：

```dockerfile
# 在 Dockerfile 中添加
RUN python -c "from sentence_transformers import SentenceTransformer; \
    SentenceTransformer('BAAI/bge-small-zh-v1.5')"
```

### 4. 监控与日志

- Prometheus + Grafana 监控服务状态
- ELK（Elasticsearch + Logstash + Kibana）收集日志
- 告警规则：响应时间 > 3s、错误率 > 5%、API 调用量异常

### 5. 备份策略

- 定期备份向量数据库（`data/vector_db` 目录）
- 备份知识库文档（`data/` 目录下的 .txt 和 .json）
- 使用云存储（S3 / OSS）进行异地备份

## 五、项目难点与解决方案

### 难点1：国内网络适配

**问题描述**：
OpenAI API 需要 VPN，HuggingFace 模型下载被墙，无法直接使用原版依赖。

**解决方案**：
1. **LLM 切换 DeepSeek**：OpenAI 兼容 API，国内直连，价格仅为 GPT-4 的 1/50
   ```bash
   export OPENAI_BASE_URL="https://api.deepseek.com/v1"
   export PRIMARY_MODEL="deepseek-chat"
   ```

2. **Embedding 本地化**：使用 `BAAI/bge-small-zh-v1.5` 在本地 CPU 运行，完全免费、无需网络
   ```python
   EMBEDDING_PROVIDER = "local"  # 默认，无需配置
   LOCAL_EMBEDDING_MODEL = "BAAI/bge-small-zh-v1.5"
   ```

3. **HuggingFace 镜像**：设置 `HF_ENDPOINT=https://hf-mirror.com` 加速首次模型下载

### 难点2：向量数据库检索精度优化

**问题描述**：
用户提问方式多样，如"多少钱"、"价格是多少"、"怎么收费"，向量检索可能无法准确匹配。

**解决方案**：
1. **相似度阈值过滤**：`SIMILARITY_THRESHOLD = 0.7`，低于阈值的结果自动丢弃
2. **Top-K 控制**：`RETRIEVAL_TOP_K = 3`，平衡检索覆盖度和噪声
3. **混合检索策略**（规划中）：关键词 BM25 + 语义向量 RRF 融合
4. **调整切分策略**：chunk_size=500 / overlap=50，按中文标点分隔

### 难点3：多轮对话上下文管理

**问题描述**：
用户使用指代词如"它支持哪些功能？"、"那价格呢？"，系统需理解上下文。

**解决方案**：
1. **对话历史注入**：将最近 3 轮对话作为上下文输入模型
2. **轮数限制**：防止上下文过长导致 token 成本和延迟增加
3. **会话超时清理**：30 分钟无活动自动过期，释放内存

### 难点4：LLM 调用稳定性

**问题描述**：
API 网络抖动或限流导致单次调用失败，影响用户体验。

**解决方案**：
1. **指数退避重试**：LLM 调用失败自动重试 2 次（2s → 4s）
2. **意图分类回退**：分类失败自动回退到 `general_chat`
3. **回答生成兜底**：生成失败返回友好提示 + 人工联系方式
4. **请求超时控制**：意图分类 15s，回答生成 60s

### 难点5：系统性能优化

**问题描述**：
向量检索和 LLM 调用耗时较长，影响用户体验。

**解决方案**：
1. **流式输出**：`self.llm.stream()` 逐 token 返回，首字延迟 < 1s
2. **Embedding 懒加载**：仅在首次向量化时加载模型，文档扫描不触发
3. **模型分级**：意图分类和回答生成可共用同一模型，减少切换开销
4. **向量缓存复用**：ChromaDB 持久化，重启无需重新向量化

### 难点6：成本控制

**问题描述**：
LLM API 调用费用随并发增长，需要有效的成本控制策略。

**解决方案**：
1. **DeepSeek 定价极低**（vs GPT-4）：
   | 指标 | DeepSeek | GPT-4 |
   |------|----------|-------|
   | 输入价格 | ¥2/百万 token | ¥110/百万 token |
   | 输出价格 | ¥8/百万 token | ¥220/百万 token |
   | 日 200 次对话月费 | ≈ ¥2 | ≈ ¥200 |

2. **本地 Embedding 完全免费**：省去远程向量化 API 费用
3. **Token 优化**：精简提示词、限制输出长度、压缩对话历史
4. **限流措施**（规划中）：单用户 QPS 限制、非工作时间降并发

### 难点7：模型兼容性

**问题描述**：
不同 LLM 厂商的 API 参数名和模型名不一致，LangChain 版本的导入路径频繁变更。

**解决方案**：
1. **统一 OpenAI 兼容层**：所有厂商 API 均使用 OpenAI 格式，通过 `OPENAI_BASE_URL` 切换
2. **显式传参**：`openai_api_base=Config.OPENAI_BASE_URL`，避免依赖环境变量名差异
3. **版本兼容导入**：try/except 兼容 langchain 0.2.x ~ 1.x 的模块路径变更
   ```python
   try:
       from langchain_core.messages import HumanMessage
   except ImportError:
       from langchain.schema import HumanMessage
   ```

## 六、故障排查

### 常见问题

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| DeepSeek API 超时 | 未设置 `openai_api_base` | 确认 `OPENAI_BASE_URL=https://api.deepseek.com/v1` |
| 模型返回空内容 | 误用推理模型 `deepseek-v4-flash` | 改用 `deepseek-chat`（非推理模型） |
| Embedding 模型下载失败 | HuggingFace 被墙 | 设 `HF_ENDPOINT=https://hf-mirror.com` |
| 知识库检索报未初始化 | 旧向量缓存维度不兼容 | 删除 `data/vector_db` 目录重建 |
| 首次运行卡住 | 正在下载 Embedding 模型（95MB） | 等待 1-2 分钟或设置 HF 镜像 |
| 检索结果为空 | 相似度阈值过高或知识库为空 | 降低 `SIMILARITY_THRESHOLD` 或检查数据文件 |
| 回答质量差 | 提示词不合适或温度太高 | 优化提示词，降低 `TEMPERATURE` |
| 内存占用高 | 向量数据库过大 | 调整 chunk_size，限制检索返回数量 |

## 七、安全建议

1. **API 密钥保护**：使用环境变量，不要硬编码或提交到 Git
2. **输入过滤**：系统已内置 `sanitize_input()` 过滤控制字符 + 超长截断（2000 字）
3. **输出过滤**：生产环境建议添加敏感信息（如内部价格、客户数据）脱敏
4. **访问控制**：API 接口添加认证中间件（JWT / API Key）
5. **日志脱敏**：日志中不记录用户完整消息内容（当前仅截取前 30 字符）
6. **定期轮换密钥**：建议每 90 天更换一次 API 密钥

## 八、扩展建议

1. **混合检索**：向量 + BM25 关键词 RRF 融合，提升召回率
2. **多语言支持**：集成翻译 API，支持多语言对话
3. **语音交互**：集成 ASR + TTS，支持语音客服
4. **情感分析**：识别用户情绪，调整回复策略
5. **知识图谱**：构建产品知识图谱，支持更复杂的推理
6. **A/B 测试**：对比不同提示词和模型的效果
7. **Redis 缓存**：缓存热门问题回答，降低 API 调用量
8. **异步架构**：FastAPI 异步 + 消息队列解耦 LLM 调用
